# RadishWeixinBot 缓存优化计划

## 一、现状分析
当前项目已有以下缓存机制：
- **轮内工具结果缓存**（`round_tool_cache`）：同一轮内相同工具+相同参数调用时复用结果
- **上下文摘要压缩**（`context_summary`）：长会话时注入摘要减少 token 消耗

## 二、缺失的缓存层

### 优先级 🔴 高 — 文件读取缓存
- **问题**：`read_file` 每次调用都从磁盘读取，同一文件在跨轮对话中被反复读取
- **方案**：在 `_run_tool` 中对 `read_file` 结果做文件级缓存，基于文件路径+修改时间
- **文件**：`llmServer/llmPolling.py`
- **预期收益**：减少磁盘 I/O，加速工具执行

### 优先级 🔴 高 — 工具文档缓存
- **问题**：`_build_user_prompt()` 每次调用 `_format_tools_docs()` 重新生成工具文档
- **方案**：将工具文档字符串缓存为实例变量，仅在工具注册变更时刷新
- **文件**：`llmServer/llmPolling.py`
- **预期收益**：减少字符串拼接开销

### 优先级 🟡 中 — LLM 语义缓存
- **问题**：每次 `sendinfo()` 调用都直接请求 API，即使完全相同的问题
- **方案**：对用户输入做 embedding 相似度匹配，对高度相似的问题直接返回缓存回复
- **文件**：`llmServer/llmPolling.py` + `llmServer/deepseek.py`
- **预期收益**：减少 API 调用次数，降低延迟和成本

### 优先级 🟢 低 — Wiki 检索结果缓存
- **问题**：`_build_wiki_context()` 每次调用都重新检索
- **方案**：对相同/相似查询缓存 Wiki 检索结果
- **文件**：`llmServer/llmPolling.py`
- **预期收益**：减少 Wiki 检索开销

## 三、实施步骤

### 第一步：实现文件读取缓存
1. 在 `Polling.__init__` 中添加 `self._file_cache = {}`
2. 在 `_run_tool` 中，当 `tool_name == "read_file"` 时，检查缓存
3. 缓存 key 为文件路径，value 为 `(mtime, result)`，文件修改后自动失效

### 第二步：实现工具文档缓存
1. 在 `Polling.__init__` 中添加 `self._tools_docs_cache = None`
2. 修改 `_format_tools_docs()` 方法，首次生成后缓存结果
3. 提供 `_invalidate_tools_cache()` 方法供工具变更时调用

### 第三步：实现 LLM 语义缓存（可选）
1. 引入 embedding 模型计算用户输入向量
2. 维护缓存数据库（可用 SQLite 或内存字典）
3. 设置相似度阈值（如 0.95），命中时直接返回缓存回复

## 四、预期效果
| 缓存类型 | 减少 API 调用 | 减少延迟 | 实现复杂度 |
|---------|:-----------:|:-------:|:---------:|
| 文件读取缓存 | 无 | 中 | 低 |
| 工具文档缓存 | 无 | 低 | 低 |
| LLM 语义缓存 | 高 | 高 | 中 |
| Wiki 检索缓存 | 无 | 低 | 低 |

## 五、注意事项
- 文件读取缓存需注意文件修改后的缓存失效
- 语义缓存需考虑 embedding 模型的选择和部署成本
- 所有缓存应支持通过配置开关启用/禁用
- 缓存大小应有限制，防止内存泄漏
